## TL; DR

Modern large language models (LLMs) demonstrate impressive reasoning abilities, yet their capacity for **long-context memory and multi-step retrieval** remains poorly understood. 

**[HashHop](https://github.com/magicproduct/hash-hop)** is a synthetic benchmark designed to isolate and stress-test these capabilities. It operationalizes the question "can this model route multi-step pointer-chasing through its context?" in a way that's resistant to the confounds that plague most long-context evals — and that question sits right at the heart of what mechanistic interpretability is trying to understand about how transformers implement algorithms.

In this short note, we play with the idea of reinterpreting HashHop not merely as a benchmark, but as a **mechanistic interpretability probe** (i.e., a behavioral interpretability tool). We try to show that Hashhop can isolate retrieval and composition circuits, eliminates semantic confounds, and enables analysis of transformer memory. 

We then walk line-by-line through the core function `MultiHopEval.make_one()` to connect implementation details to interpretability insights.

## Why HashHop is an interesting probe tool for interpretability

> *While traditional benchmarks like "Needle in a Haystack" test if a model can find a specific fact in a long document, HashHop is significantly more difficult because it tests a model's ability to maintain a chain of reasoning over millions of tokens without relying on semantic shortcuts.*

Most benchmarks conflate multiple abilities:

* language understanding
* world knowledge
* reasoning heuristics

HashHop removes all of that.

It replaces meaning with **random hashes**, forcing the model to rely purely on:

* memory
* attention-based retrieval
* compositional chaining

Furthermore, in the context of interpretability, perhaps we can draw the following connections:

#### 1. HashHop as a probe for induction heads (i.e., "locating the memory")

Mechanistic interpretability has identified induction heads — attention heads that implement a simple "find the previous token, copy what came after it" algorithm. HashHop is essentially a multi-hop generalization of exactly this circuit. Each hop requires the model to locate a hash string in context and copy its successor. So HashHop performance is, implicitly, a behavioral measure of whether induction-like circuits can chain together across long distances. Interpretability would ask: are these hops being resolved by the same heads in series, or does the model learn a separate "multi-hop" circuit?

#### 2. The chain-of-thought condition is a scratchpad experiment

The CoT vs. no-CoT split in HashHop maps directly onto the interpretability question of where computation happens. Without CoT, the model must resolve multiple hops in residual stream "silence" — all intermediate state has to be stored in activations. With CoT, intermediate hashes are written to the context, giving the model external memory. Interpretability researchers (e.g., the work on "faith and fate" in reasoning) care a lot about whether transformers are doing multi-step reasoning internally or offloading it to the context. HashHop gives a clean behavioral signal for exactly this.

#### 3. The incompressibility property targets superposition

**[Toy Models of Superposition (Elhage et al., Anthropic)](https://arxiv.org/abs/2209.10652)** showed that models represent more features than they have dimensions by superposing them. Semantic needles in hay are dangerous for evaluation because a model might recover the answer via a compressed, high-level feature rather than actual retrieval. Random hashes are deliberately outside the model's feature space — they can't be compressed or reconstructed from prior knowledge. This means HashHop failures can't be attributed to superposition artifacts or feature bleed. It isolates the retrieval circuit cleanly.

#### 4. Performance degradation with hop count maps to circuit depth

The finding that models degrade with more than 2 hops (without CoT) is interpretability-relevant. It suggests a depth ceiling on whatever internal circuit is doing the retrieval — consistent with the idea that multi-layer sequential computation is hard to route through attention because information has to survive in the residual stream across multiple layers without a dedicated "working memory" mechanism. This is essentially a behavioral signature of the limitations Anthropic and DeepMind researchers have found when probing for multi-step reasoning circuits mechanistically.

#### 5. It's a natural complement to activation patching

If you wanted to do mechanistic work on a model using HashHop, the setup is ideal: the input structure is perfectly controlled (you know exactly where each hash pair is), the task is unambiguous, and you can vary difficulty on a single axis (hop count). This makes it easy to design activation patching or causal tracing experiments — patch in the embedding of H2 at position X, see if the model now correctly resolves H2 → H3. That's much harder to do with semantically messy needles.


### HashHop's core task

Given:

```
A → B
B → C
```

Query:

```
A → ?
```

Answer:

```
C
```

This is a **multi-hop retrieval problem**.

---

## Mechanistic framing

HashHop directly probes a specific internal circuit:

```
lookup(A → B)
then lookup(B → C)
then output(C)
```

This maps cleanly to interpretability concepts:

| Step        | Mechanism                        |
| ----------- | -------------------------------- |
| Find A → B  | Retrieval heads                  |
| Store B     | Working memory (residual stream) |
| Query B → C | Second retrieval                 |
| Output C    | Readout layer                    |

---

## What makes HashHop fun

### 1. No semantic shortcuts

* Tokens are random strings
* No compression via meaning
* No prior knowledge

👉 Success implies *true retrieval mechanisms*


### 2. Clean separation of abilities

| Setting      | What it tests           |
| ------------ | ----------------------- |
| 1-hop        | Pure retrieval          |
| 2+ hops      | Compositional reasoning |
| Long context | Memory scaling          |


### 3. Traceable failures

When models fail, they typically:

* retrieve incorrect intermediate nodes
* stop after one hop
* attend to nearby distractors

These are **mechanistically interpretable errors**.

---

## Mental model of a transformer solving HashHop

A transformer solving HashHop likely does:

1. Scan context → build implicit index
2. Query A → retrieve B
3. Store B in residual stream
4. Query B → retrieve C
5. Output C

Interpretability questions:

* Where is the index stored?
* Which heads perform retrieval?
* How is B preserved across layers?

## Code Walkthrough: `MultiHopEval.make_one()`

This function generates a **single evaluation instance**.

---

### Step 1: Parameterization

Conceptually, the function takes inputs like:

```python
n_chars_problem
num_queries
hops
hash_pair_str_length
chain_of_thought
```

#### Interpretability meaning

* `n_chars_problem` → **context length stress test**
* `hops` → **depth of compositional circuit**
* `num_queries` → **batch evaluation of retrieval**
* `hash_pair_str_length` → **token entropy / collision resistance**

---

### Step 2: Generate random hashes

```python
def random_hash():
    return ''.join(random.choices(alphabet, k=hash_pair_str_length))
```

This creates identifiers like:

```
XJf93KlaQp
```

#### Interpretability insight

* Removes semantic priors
* Forces exact-match retrieval
* Prevents embedding-space shortcuts

---

### Step 3: Build chains

```python
chains = []
for _ in range(num_queries):
    chain = [random_hash() for _ in range(hops + 1)]
    chains.append(chain)
```

Example (2-hop):

```
[A, B, C]
```

#### Interpretability insight

Each chain is a **ground-truth reasoning path**:

```
A → B → C
```

This is the circuit we want the model to reconstruct.

---

### Step 4: Convert chains into pairs

```python
pairs = []
for chain in chains:
    for i in range(len(chain) - 1):
        pairs.append((chain[i], chain[i+1]))
```

Result:

```
(A → B)
(B → C)
```

#### Interpretability insight

This step **decomposes reasoning into local edges**.

The model must:

* reconstruct the graph
* perform traversal at inference time

---

### Step 5: Add distractors

```python
while len(pairs) < target_size:
    pairs.append((random_hash(), random_hash()))
```

#### Interpretability insight

This introduces:

* noise
* interference
* attention competition

Now the model must:

* filter irrelevant edges
* avoid spurious matches

This is critical for testing **selective attention mechanisms**.

---

### Step 6: Shuffle context

```python
random.shuffle(pairs)
```

#### Interpretability insight

Destroys:

* locality
* ordering cues

Forces:

* global retrieval
* true search over context

---

### Step 7: Build prompt

```python
prompt = "\n".join(f"{a} = {b}" for a, b in pairs)
```

Example:

```
X1 = Y7
Q9 = Z3
A2 = B8
...
```

#### Interpretability insight

This becomes the **external memory store**.

The model must internally:

* index these mappings
* retrieve them via attention

---

### Step 8: Create queries

```python
queries = [chain[0] for chain in chains]
```

Each query is:

```
A → ?
```

#### Interpretability insight

This is the **entry point** into the retrieval chain.

---

### Step 9: Compute targets

```python
targets = [chain[-1] for chain in chains]
```

#### Interpretability insight

Defines the **correct multi-hop output**.

---

### Step 10: Optional chain-of-thought

```python
if chain_of_thought:
    completion = "A → B → C"
else:
    completion = "C"
```

#### Interpretability insight

This toggles:

* **explicit reasoning traces**
* vs **implicit internal computation**

Useful for studying:

* whether reasoning is latent or verbalized

## Mechanistic interpretation of the full pipeline

Putting it all together:

### Input

* Large set of unordered key-value pairs

### Internal process (hypothesized)

1. Attention heads scan for matches to query token
2. Retrieve first mapping (A → B)
3. Store B in residual stream
4. Re-query using B
5. Retrieve (B → C)
6. Output C

## Closing Thought

Most evaluations ask:

> "Can the model give the right answer?"

HashHop asks:

> "Can the model *find* the answer inside its own context?"

That shift — from output correctness to **internal mechanism** — is what makes it interesting for interpretability studies.
